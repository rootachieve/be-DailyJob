### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 공통된 필드가 많지만, 다른 정보를 가지는 두가지 경매 종류를 어떻게 표현할지 고민하였다.
  - 팀원과 논의하다 table per class 를 도입하여 이를 표현하였다.
- 그렇게 판단한 이유:
  - 각 테이블의 중복되는 값도 많지만 다른 값도 많다. 그렇기에 나누긴 해야하지만, 공통 테이블을 두면 검색 결과등에서 join을 한 뒤 표현해야하기에 비용이 크다 생각했다.
- 고려했지만 선택하지 않은 대안:
  - 하나의 테이블로 통합한다. -> 겹치지 않는 값도 많기에 null이 많아지고 확장시 문제가 생길 수 있었다.
  - 공통 테이블을 두고 각 경매 타입의 테이블에서 참조하도록 하여 Join 한다 -> 전체 경매 조회시 Join의 비용이 커지게 된다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - join을 하지 않고 각 테이블을 표현할 수 있기에 표현이 간편해진다.
  - 하지만 공통 필드의 중복이 많기에 추후 새로운 경매가 추가된다면 수정이 필요하다.
  - 이미지 등 경매를 외래키로 가지는 테이블이 두캐의 키를 가지는 것을 막기위해 경매 테이블들은 TSID를 지닌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 경매시스템에서 판매자 혹은 구매자의 노쇼가 발생하였을때, 보증금과 이후 후속 대응에 대해 고민하였다.
- 처음 생각한 방안
  - 보증금을 100%으로 하여 구매자 노쇼시 그대로 판매자에게 가도록 하고, 판매자 노쇼시 계정을 정지시키는 방안을 생각했다.
  - 하지만 이 경우 보증금을 노린 행위가 생길 수 있고, 구매자 입장에서 100%를 전부 보증금으로 넣어두기 부담스러울 것 같다는 판단을 하였다.
- 선택 방안
  - 보증금을 20%로 두고, 이후 나머지 80% 금액을 입금하도록 하는 방법을 선택하였다.
  - 하지만 이 경우도 노쇼에 대한 대응이 부족하기에 추후 더 논의 해보려한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 막히는 문제를 계속 고민하기 보다 밥을 먹거나, 산책을 한 뒤 다시 확인하였다.
- 그 접근이 도움이 된 이유:
  - 한번 멀리 떨어진 뒤 다시 바라봄으로서 문제를 좀 더 다양한 시각으로 바라볼 수 있었다.
  - 그 결과 이전보다 더욱 집중하며 더욱 나은 결과를 만들어 낼 수 있었다.
- 다음에도 반복하고 싶은 점:
  - 집중 시간을 정해두고, 그 시간동안은 확실히 집중하고 잠시동안 산책을 하거나 쉬어보려한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 기획을 하는 과정에서 ERD, 기획, 기능 명세, 업무 배분등의 여러 부분을 왔다갔다하며 작업하느라 제대로 수행하지 못했다.
- 그렇게 된 원인:
  - 극 초기이기에 작업이 배분되지 못하고, 교착상태가 되는 경우가 많았다.
  - 그렇기에 이를 환기하기위해 다른 작업으로 전환하였지만, 이전 작업이 다 완료되지 못하였기에 다시 도중에 다른 주제로 빠지는 경우가 많았다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 하나의 작업을 온전히 수행한 뒤 다음 작업을 수행한다.
  - 하지만 그러면서도 너무 하나의 주제에 집중되는것을 막기위해 시간을 정하고 토론한 뒤, 결정하고 넘어간다.

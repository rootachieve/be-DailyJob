### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - Concurrent 패키지의 기능을 서비스에 적용하는 과정에서 가상 스레드를 사용해보았다.
- 그렇게 판단한 이유:
  - 현재 서비스는 다른 기능없이 요청에 따라 파일을 읽어 반환하는 서비스이다. 
  - 즉 IO가 많기에 그냥 스레드보다 가상 스레드를 사용한다면 IO 도중 플랫폼 스레드를 재사용하여 처리량을 높일 수 있으리라 판단했다.
- 고려했지만 선택하지 않은 대안:
  - 고려사항에 스레드 풀이 있었다.
  - 매번 스레드를 만드는 대신 스레드의 수를 제한하여 문맥 전이 늘어나는 것을 막고 일관된 서버 부하를 만들어낸다.
  - 하지만 현재 코드 같이 IO 기반인 경우 느린 몇개의 요청이 스레드 풀을 차지하면 다른 요청이 대기해야하기에 비효율적이라 생각했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 가상 스레드가 이렇게 보면 다 좋아보이지만 이후 다른 분들과 이야기 하던 도중 가상 스레드 사용시 리소스나 플랫폼 스레드 관련하여 조심할 부분이 있다는 것을 알게되었다.
  - 이부분은 더 공부하여 어떤 부분을 조심해서 사용해야할지 탐구해보려한다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 직접 겪은 문제는 아니지만, ERD에 대해 교육을 듣고, 다이어그램을 작성해보던 도중 최근 학습한 Spring JPA와 관련하여 아래와 같은 의문이 생겼다.
  - > 이렇게 데이터 중심으로 모델링을 한다면 서로 멀리 있는 데이터가 필요하다면 JPA에서 계속 객체를 타고 가야하는게 아닌가?
- 처음 추측한 원인:
  - 학습한 내용으로는 Spring JPA에서 페치타입에 Lazy를 걸어두면 외래키에 해당하는 객체는 해당 객체에 접근할때 SQL 쿼리를 통해 가져온다.
  - 즉 멀리 있는 데이터에 접근하고자 하면 계속 SQL 쿼리를 보내며 객체를 타고 건너가야하고, 이는 바람직하지 못하다 생각했다.
  - Join을 하기에도 관련있는 테이블이 많아진다면 비용이 클것이다.
  - 그렇다면 비용을 줄이기 위해선 모델링을 할때 객체를 계속 타고 넘어가는 형태를 줄여야한다.
  - 결국 성능을 고려한다면 모델링 단계에서부터 객체를 계속 타고 이동해야 하는 구조를 줄이는 것이 필요하다.
- 원인을 확인하기 위해 시도한 방법:
  - 주변 분들에게 물어보고 AI와 토론해보았다.
- 결론
  - 이러한 문제의식은 DDD의 애그리거트 개념과 유사하다는 것을 알게 되었다. 
  - DDD에선 모든 테이블을 객체 연관관계로 두기 보단 하나의 애그리거트 내부에서만 객체 참조를 자연스럽게 사용하고, 다른 애그리거트는 ID 참조를 고려한다.
  - 즉 ERD의 외래키 관계를 JPA 객체 관계로 1:1 매핑하는 것이 항상 좋은 설계는 아니며, 도메인 규칙을 기준으로 객체 간 참조 범위를 정해야 한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - MIME를 타입을 처리하는 과정에서 Accept 값을 기준으로 Content-Type을 선택하려 했지만, 실제 구현 난이도와 서비스 상황을 충분히 고려하지 못했다.
- 그렇게 된 원인:
  - Accept 헤더는 하나의 MIME 타입만 전달되는 것이 아니고, 거기에 더해 q값에 따라 알맞은 타입을 결정해야해 생각보다 복잡한 처리가 필요했다.
  - 반면 현재 서비스에서 제공하는 응답 타입은 제한적이었기 때문에, 모든 Accept 조건을 고려하는 것은 서비스가 얻는 이점에 비해 구현 비용이 큰 접근이었다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 개발 이전 구현 비용과 서비스의 이점을 잘 고려하여 필요한 수준까지 적용하는 것이 중요함을 느꼈다.

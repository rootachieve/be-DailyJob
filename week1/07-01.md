###### 2. 오늘 가장 의미있었던 기술적 판단
> Http Request를 받고, 알맞은 파일을 응답하는 것이 `RequestHandler` 의 책임이다. 
> 그렇다면 요청을 읽고, Url을 추출하며, 필요한 정보를 로깅하는 것은 별도의 클래스(책임)으로 빼야하는게 아닐까?
- 내가 내린 판단:
  - 아직 분리하지 않고, 추후 http Request에 대한 요구사항이 늘어나면 분리한다.
- 그렇게 판단한 이유:
  1. 현재까지 RequestHandler에는 요청 관련 로직이 아직까진 짧은 정도로 존재함.
  2. 또한 Request 자체 혹은 헤더에 대한 기능적 요구가 많지 않음.
  3. 다른 핸들러 존재하지 않기에 추출시 중복코드 감소 없음.
- 고려했지만 선택하지 않은 대안:
  - HttpRequest에 관련된 메서드, 필드를 HttpRequest 클래스로 분리하여 모은다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 불필요한 클래스, 추상화를 하지 않아 한 곳에서 코드를 파악할 수 있다.
  - 한편으론 파서, 추출 등의 로직을 개별적으로 테스트하긴 어렵다(private).
    - 테스트하고자 한다면 protected, 패키지 private하게 해야한다!
  - 추후 body, 쿠키와 같이 더 복잡한 로직이 추가되면 핸들러의 책임이 커지기에 분리해야한다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - Http 요청 헤더를 분석하기 위해 먼저 inputStream을 라인별로 읽어 출력하도록 하고 실행해 보았는데 `Broken pipe` 에러가 발생했다.
- 처음 추측한 원인:
  - 잘못된 방식으로 스트림을 읽어 객체 생성도중 무언가 충돌한 것이라 생각하였다.
- 원인을 확인하기 위해 시도한 방법:
  - 객체 생성 단계를 분리하여 디버깅 포인트를 찍으며 확인해보았다.
- 실제 원인:
  - 모든 줄을 읽기 위해 `while ((line = buf.readLine()) != null) { //...`과 같이 입력을 읽었었는데, HTTP 클라이언트는 응답을 받기전까지 연결을 유지한다.
  - 즉, 응답을 받기전까지 클라이언트가 닫히지 않고, null 상태가 되지 않는다.
  - 이후 타임아웃으로 연결이 끊기고, 그때야 응답하려하기에 에러가 생긴다.
- 해결 방법:
  - Http 헤더는 빈줄로 끝나기에 조건에 공백 검사를 추가한다.
  - `while ((line = reader.readLine()) != null && !line.isEmpty()) {`
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 넘어오는 데이터의 형식을 확인하여 종료조건을 확실하게 확인하기.

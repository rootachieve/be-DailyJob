### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 범용 SQL 파서를 구현하는 대신, 기존 SQL과 일대일로 대응하는 `GET`, `PUT`, `SET` 키워드 기반의 간단한 쿼리를 도입하였다.
- 그렇게 판단한 이유:
  - 현재 서비스에서 사용하는 SQL의 종류가 한정되어 있어 범용 문법을 지원할 필요가 없었다.
  - 쿼리에는 명령과 테이블, 조건 컬럼만 두고 실제 값은 별도 인자로 전달하면 SQL을 직접 치환하지 않고도 필요한 기능을 표현할 수 있었다.
- 고려했지만 선택하지 않은 대안:
  - SQL 혹은 쿼리를 범용적으로 파싱하는 방식 -> 구현해야 할 문법과 예외가 너무 많아 사실상 엔진을 만드는 일이 되었다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 파싱 로직과 쿼리 문자열이 짧아져 현재 서비스에서 사용하는 동작을 쉽게 확인할 수 있게 되었다.
  - 범용 SQL은 지원하지 않지만, 새로운 동작이 필요해질 때 해당 서비스 쿼리만 명시적으로 추가하면 된다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - CSV 저장소에서도 기존 DAO의 SQL과 `PreparedStatement` 사용 방식을 유지하려 했지만, SQL을 어디까지 어떻게 해석해야 할지 막혔다.
- 처음 추측한 원인:
  - SQL의 `?`를 전달받은 값으로 치환한 뒤 문자열을 파싱하여 실행하면 충분히 처리할 수 있다 판단하였다.
- 원인을 확인하기 위해 시도한 방법:
  - 서비스에서 실제로 사용하는 `SELECT`, `INSERT`, `UPDATE` 쿼리를 모두 나열하고 필요한 조건, 정렬, 제한, `IN` 문법을 비교하였다.
  - 이후 해당 키워드들을 바탕으로 로직을 적용하는 구상을 하였다.
- 실제 원인:
  - `?`를 문자열로 직접 치환하면 SQL 구문과 값의 경계가 사라져 SQL 인젝션을 안전하게 막을 수 없었다.
  - 구문 해석과 값 바인딩을 분리하기위해선 파싱을 하면서도 값과 키워드가 분리되고 그 의미를 가져야했는데 이렇게 된다면 결국 AST에 가까운 파서가 필요했다.
  - 과제의 목적은 SQL 엔진 구현이 아니라 현재 애플리케이션을 CSV 저장소에서 동작시키는 것이었는데, 문제의 범위를 지나치게 넓게 잡고 있었다.
- 해결 방법:
  - 범용 SQL 지원을 포기한 뒤 고민하다 현재 서비스의 SQL과 일대일로 대응하는 쿼리만 정의하였다.
  - 여러 시도로 시간이 부족해진 뒤 지원 범위와 쿼리 형식을 구체적으로 정리해 AI에 전달하고, 구현 결과를 JDBC 객체와 DAO에 연결하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 범용 기능을 설계하기 전에 현재 호출부가 실제로 요구하는 입력과 동작을 먼저 전부 나열한다.
  - 직접 구현해야 하는 학습 목표와 과제에서 단순화해도 되는 범위를 먼저 구분한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 잘모르는 주제를 학습하는 과정에서 AI에게 내가 이해한바를 설명하고 검증받도록 하였다.
- 그 접근이 도움이 된 이유:
  - 내가 설명을 함으로서 내가 이해한 바를 정리하고 다시 한번 생각해볼 수 있도록한다.
  - AI가 바로 잘못된 부분을 잡아주도록 하여 내 지식의 공백, 잘못된 지점을 매꿀 수 있었다.
  - 또한 AI가 더 나아가야할 주제를 추천해주도록 하여 학습을 이어나갈 수 있었다.
- 다음에도 반복하고 싶은 점:
  - 앞으로 배우게될 새로운 주제도 비슷하게 학습하며 나아가보려한다. 
  - 5번 항목과 관련되어 먼저 작업이전에도 작업에 대한 내용을 설명하며 점검 받아보려한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 서비스가 사용하는 쿼리를 먼저 한정하지 않고 범용 SQL과 별도 쿼리 파서를 차례로 고민하면서 많은 시간을 사용하였다.
- 그렇게 된 원인:
  - 처음부터 기존 SQL을 최대한 유지해야 한다고 생각해 과제에서 실제로 필요한 범위보다 문제를 크게 잡았다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 구현 전 지원해야 하는 쿼리 목록과 지원하지 않을 기능을 먼저 적고, 그 범위에서 가장 단순한 표현을 선택한다.
  - 범용 구현을 시도할 때는 탐색 시간을 정해두고, 범위를 벗어나면 현재 서비스에 필요한 기능으로 빠르게 줄인다.
  - AI가 구현한 결과는 처음 정리한 요구사항과 대조하고, 불필요하게 범용화된 코드는 바로 단순화한다.

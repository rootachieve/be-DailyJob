### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 어노테이션이나 리플렉션으로 자동 주입하는 대신 생성자 기반 DI를 사용하고, ApplicationConfig 한 곳에서 애플리케이션과 WAS를 조립하도록 하였다.
  - Logic은 HttpRequest/HttpResponse를 모르도록 하고, 요청 파싱과 응답 조립은 Apis와 WAS의 책임으로 두었다.
- 그렇게 판단한 이유:
  - 어노테이션과 리플렉션으로 클래스를 찾아 자동으로 조립하는 방식은 지금 과제의 범위에 비해 부담이 컸다.
  - 애플리케이션이 WAS를 알아야 하는 결합지점이 하나 생기더라도, MainApplication과 ApplicationConfig에만 모으면 나머지 코드의 경계는 명확하게 유지할 수 있다고 판단하였다.
  - Logic이 HTTP 객체를 반환하면 비즈니스 로직의 입력과 출력이 웹 요청에 제한되기 때문에, 값이나 예외만 반환하는 편이 다른 환경에서도 재사용하고 테스트하기 좋다고 생각하였다.
- 고려했지만 선택하지 않은 대안:
  - Apis와 Router를 static으로 연결하는 방법 -> 전역 상태와 직접 의존이 그대로 남는다 판단
  - 프록시를 두어 static 의존을 간접화하는 방법 -> 실제 결합은 남은 채 구조만 복잡해진다 판단
  - 어노테이션과 리플렉션으로 클래스와 메서드를 찾아 자동 조립하는 방법 -> 현재 단계에서 구현 비용이 너무 크다 판단
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 각 객체가 필요한 의존성을 생성자로 드러내게 되어 테스트에서 필요한 구현을 직접 전달할 수 있게 되었다.
  - WAS는 비즈니스 예외와 애플리케이션의 응답 정책을 모르고, 애플리케이션은 WAS 위에서 사용할 Router와 ExceptionHandler만 조립하는 경계가 만들어졌다.
  - Logic 테스트에서 HttpRequest/HttpResponse를 만들 필요가 줄어 비즈니스 규칙을 값 중심으로 확인할 수 있게 되었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - WAS와 애플리케이션 패키지를 나눈 뒤 예외를 애플리케이션의 HTTP 응답으로 바꾸는 책임을 어디에 두어야 할지 애매했다.
- 처음 추측한 원인:
  - 요청 파싱과 응답 처리가 WAS의 책임이므로 기존처럼 WAS의 ExceptionHandler가 예외를 처리하는게 맞다는 판단하였다.
  - 하지만 이 경우 애플리케이션 레이어의 규격화된 응답을 보낼 수 없었다(의존성)
- 원인을 확인하기 위해 시도한 방법:
  - BusinessException이 발생하는 위치부터 예외가 전달되는 흐름을 따라가며 각 계층이 알아야 하는 타입을 확인하였다.
  - 애플리케이션의 예외 처리 규칙을 WAS 내부에 두는 방법과, WebServer를 만들 때 외부에서 전달하는 방법을 비교하였다.
- 실제 원인:
  - 예외를 잡는 시점은 요청 전체를 감싸는 RequestHandler가 적절하지만, BusinessException을 어떤 상태 코드와 리다이렉트 응답으로 바꿀지는 애플리케이션의 정책이었다.
  - 예외를 잡는 위치와 예외를 응답으로 해석하는 책임을 같은 것으로 생각하여 경계가 모호했던 것이다.
- 해결 방법:
  - WAS에는 ExceptionHandler 인터페이스만 남기고, 예외를 실제 HTTP 응답으로 바꾸는 ApplicationExceptionHandler를 애플리케이션에 두었다.
  - ApplicationConfig가 WebServer를 생성할 때 ApplicationExceptionHandler를 함께 전달하고, RequestHandler가 전체 요청 처리 흐름을 감싸도록 변경하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 코드가 실행되는 위치와 정책을 결정하는 주체가 같은지 먼저 구분한다.
  - 계층을 분리할 때 패키지 위치만 옮기지 않고, 하위 계층이 상위 계층의 타입을 알고 있는지 의존 방향까지 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - static, 프록시, 리플렉션 같은 대안을 바로 구현하지 않고 각각 어떤 의존을 남기는지 고민한 뒤 생성자 기반 DI를 선택하였다.
  - 애플리케이션 레이어를 분리한 뒤에도 Logic의 HTTP 의존과 WAS의 BusinessException 의존을 계속 찾아 경계를 한 단계씩 정리하였다.
- 그 접근이 도움이 된 이유:
  - 패키지만 나누는 데서 끝나지 않고, 실제 객체 생성과 요청/응답 및 예외의 흐름까지 분리할 수 있었다.
  - 처음부터 자동 DI를 만들었다면 과제의 핵심보다 기반 코드 구현에 더 많은 시간을 썼을 텐데, 생성자 조립으로 필요한 수준만 구현할 수 있었다.
- 다음에도 반복하고 싶은 점:
  - 구조를 변경할 때 클래스의 위치뿐 아니라 생성, 호출, 예외 전달 흐름을 끝까지 따라가며 경계를 확인한다.
  - 구현이 끝난 뒤 diff와 전체 흐름을 다시 리뷰하여 큰 변경 중 놓친 세부 문제를 점검한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 처음 구조를 변경할 때 일부 API 경로에 application 접두사가 들어가고, Logic이 다른 Logic을 거쳐 DB에 접근하는 의존과 중복 에러 코드가 남은 것을 바로 발견하지 못했다.
- 그렇게 된 원인:
  - 간단한 패키지 이동과정에서 테스트까지 이동하기 번거로워 AI를 이용하였는데, 이때 AI가 지시를 잘못 이해한 듯하다.
  - static 의존 제거와 패키지 이동 범위가 커서 객체가 정상적으로 조립되는지에 집중했고, 기존 URL과 세부 의존 방향을 함께 확인하지 못하였다.
- 다음에는 다르게 해볼 구체적인 행동:
  - AI에게 지시를 좀 더 명확하게 내리고, AI가 작성한 코드를 아무리 바쁘더라도 검증한다.
  - 큰 구조 변경 전에는 유지되어야 하는 외부 동작(API 경로 등)과 의존 방향을 짧은 체크리스트로 먼저 적고, 변경 후 각각 확인한다.

> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 웹서버 9단계 구현을 위한 구조 리팩터링
    - WAS 예외를 `HttpException` 계층으로 통합하고 오류 표현 정책을 `ApplicationExceptionHandler`로 이동
    - 인증과 라우팅 fallback 전략을 외부에서 주입하도록 변경
    - `DataBaseManager`가 `DataSource`를 주입받도록 하고 조회, 수정, 생성 명령을 분리
    - DB 접근 오류를 `DataAccessException`으로 분리하고 테이블 초기화를 애플리케이션 시작 시점으로 이동
    - JDBC 관련 코드와 이미지 저장 코드를 `infrastructure` 패키지로 이동
    - 쿼리 파싱과 테스트 데이터베이스 구성을 정리
  - JDBC의 `Driver`, `Connection`, `PreparedStatement`, `ResultSet` 흐름과 트랜잭션 및 커넥션 풀의 역할 학습
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 웹서버 9단계

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 리팩토링 과정에서 AI가 Database를 인터페이스 등으로 두고, 구현체를 만들자 하였지만, 과한 추상화라 생각하여 적용하지 않았다.
- 그렇게 판단한 이유:
  - 현재 Database는 교체 대상이 아니고, 단 하나만 존재하며, 테스트시에도 테스트 DB를 만드는 행위로 충분히 테스트 가능하다.
  - 그렇기에 더 복합적인 테스트를 하거나, 교체 대상이 되는 경우, 혹은 이 구현을 가릴 필요가 있어질 경우에 그때, 추상화를 적용하면 될 것이라 판단했다.
  - 이와 마찬가지로 Authenticator도 전략이지만 아직 다른 전략이 없기에 추상화하지 않았다.
- 고려했지만 선택하지 않은 대안:
  - 인터페이스를 정의하고 운영용/테스트용 구현체를 분리하는 방식 -> 실제 테스트용이 아직 필요하지 않다. 
  - 의존성 주입을 통해 구현체를 외부에서 교체할 수 있도록 만들기 -> 아직 교체 대상이 없기에 아직 필요하지 않다.
  - Authenticator를 전략 인터페이스와 단일 전략 구현체로 분리 -> 마찬가지로 아직 교체 대상이 없다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 불필요한 계층과 보일러플레이트가 줄어 코드 흐름이 단순해졌다.
  - 테스트가 실제 DB 동작까지 검증하기에 신뢰도가 높지만, 실행 속도와 테스트 격리 비용은 조금 커질 수 있다.
  - 당장의 변경은 줄였지만, 추후 변경이 필요해질때 인터페이스를 추출하는 비용이 발생한다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - `ImageStorage`, DB 코드와 앞으로 만들 JDBC 드라이버를 어느 레이어에 두어야 할지 고민이였다.
- 처음 추측한 원인:
  - 애플리케이션이 저장 위치를 결정하면 안 된다고 생각해 이 기능들을 WAS의 책임으로 보았다.
- 원인을 확인하기 위해 시도한 방법:
  - 각 코드가 무엇을 하는지가 아니라 어떤 이유로 변경되는지를 기준으로 애플리케이션, WAS의 책임을 다시 비교하였다.
- 실제 원인:
  - 그러자 ImageStorage와 DB등은 애플리케이션도, WAS도 아닌 별도의 기준을 지닌다는 것을 알게되었다.
  - 애플리케이션과 WAS만으로 모든 코드를 나누려 해 저장 기술이라는 변경 축을 놓치고 있었다.
- 해결 방법:
  - JDBC 관련 코드는 `infrastructure.persistence.jdbc`, 이미지 저장은 `infrastructure.storage`로 이동하도록 하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 코드의 실행 위치보다 변경 원인이 같은 코드끼리 모여 있는지 먼저 확인한다.
  - 패키지 구조 작성시 패키지끼리의 의존성 또한 깊게 고려한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - AI를 통해 코드리뷰를 진행할때, 두 AI(클로드, 코덱스)를 동시에 돌리고, 서로 교차 검증하도록 하여, 최종 코드리뷰 완성본을 만들어 냈다.
- 그 접근이 도움이 된 이유:
  - 이렇게함으로서 하나의 방향에 집중되어 발생할 수 있는 누락, 과한 개선 등을 방지할 수 있었다.
  - AI가 서로 피드백을 하고 잘못된 방향을 짚어줌으로서 더 나은 코드 품질을 만들어 낼 수 있었다.
- 다음에도 반복하고 싶은 점:
  - AI를 서로 검증하도록 사용하여 확실한 판단을 내리는데 도움이 되도록 한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 이런 구조 개선, 여러 원칙을 미리 미리 고려해야했지만, 과제 구현으로 인해 놓친 부분이 좀 있던 것 같다.
- 그렇게 된 원인:
  - 과제를 하면서도 구조, 추상화, 책임등을 많이 고려하였지만, DI와 인터페이스, 전략등 주입하는 형태를 오늘 고려하기 시작하여 발생한 듯하다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 구현을 하면서도 확장, 변경가능성을 따져본뒤, 미리 인터페이스, 전략 등을 주입하는 구조를 작성한다.
  - 하지만, 그러면서도 모든 것을 추상화하거나, 과하게 작성하지 않기위해 노력한다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 인증, 인가 예외 상황에서 다른 화면으로 이동시킬 때 302 상태 코드를 리다이렉트로 사용하던 것을 제거 하고 자바스크립트 실행 기반 리다이렉트로 변경
- 그렇게 판단한 이유:
  - 302는 리소스가 이동했다는 의미이지 예외 상황을 표현하는 상태 코드가 아니다.
  - 그렇다고 401만 던지면 지금 구조상 다른 화면으로 이동시킬 가 없었다.
  - 스쿼드 세션에서 응답 바디에 자바스크립트를 담으면 그대로 실행된다는 점을 확인한 뒤, 이를 사용하면 예외 표현과 리다이렉트 모두 가능할 것이다 판단했다.
- 고려했지만 선택하지 않은 대안:
  - html을 수정하여 400, 401 혹은 에러코드를 보고 리다이렉트: 기존의 흐름과 다른 로직을 추가해야하는 것이기에 바람직 하지 않다 판단
  - 기존처럼 302를 리다이렉트로 사용: 상태 코드의 의미를 잘못 사용한다 판단
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - `ErrorCode`마다 상태 코드와 리다이렉트 경로를 함께 선언하게 되어, 예외가 늘어나도 각 API가 리다이렉트 경로를 직접 만들 필요가 없어졌다.
  - 여러 곳에 흩어져 있던 302 기반 리다이렉트 코드를 걷어내면서 예외 처리 경로가 하나로 좁혀졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 게시글 작성에 이미지 업로드를 붙이려는데, 이미지를 어디에 저장하고 어떤 식별자로 다시 보여줄지 판단이 서지 않았다.
- 처음 추측한 원인:
  - 이미지를 미리보기로 보여주기 위해선 현재 html은 정적이기에 이미지를 먼저 저장해야한다 판단했다.
  - DB저장과 서버 로컬에 저장하는 방식중 로컬에 저장하는 것이 정적리소스를 가져오는 측면에 맞을 것 같아, 이미지 업로드시 서버 로컬에 저장하고, 정적 html을 내려주고자 했다.
- 원인을 확인하기 위해 시도한 방법:
  - 만약 사용자가 이미지를 업로드만 하고, 글 작성을 안하면 그 이미지는 참조되지 않으면서 자원만 차지한다.
- 실제 원인(막힌 지점):
  - 사용자가 선택한 이미지를 js(과하지 않은 정도)를 사용하면 업로드 없이 미리보기로 표현이 가능하다는 것을 검색을 통해 알게되었다. 
  - 이와 별개로 순차 증가 id나 게시글에 묶인 id를 쓰면 그 값을 아는 사람은 누구나 다른 사람 이미지에 접근할 수 있다.
- 해결 방법:
  - 이미지 식별자를 추측 불가능한 UUID로 정하고, 파일시스템에 `UUID.jpeg`로 저장하고, url을 참조하도록 하였다.
  - 이미지 업로드가 필요없어 졌기에 한번에 폼 데이터를 보내는 멀티파트를 학습한 뒤. 이미지와 컨텐츠를 멀티파트로 보내도록 하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 식별자를 설계할 때 이 값을 안다면 무엇까지 접근할 수 있는지를 먼저 따져보고 순차 증가 id 사용 여부를 결정한다.
  - 너무 시야를 좁게 생각하지말고 문제를 멀리서 넓게 바라보며, 더 쉬운 해결법을 찾아본다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:

- 그 접근이 도움이 된 이유:

  - 다음에도 코드 구현도 구현이지만, 내가 고민했던 요소, 생각을 공유하며 행여 놓친부분을 잡거나, 더 나은 방법을 찾아보려한다.

### 6. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:
>현재 코드는 ExceptionHandler가 Route에 존재하여 API에서 발생하는 예외를 최종적으로 잡는 역할을 합니다. 그리고 그 앞단에 HttpReqeust를 파싱하는 과정에서 발생하는 예외를 잡기위한 try, catch가 하나더 존재합니다.
저는 처음에는 ExceptionHandler를 앞단으로 빼 HttpRequest의 예외까지 한번에 잡는걸 생각했습니다. 하지만 지금 다시 생각해보니 ExceptionHandler는 API의 에러를 잡는 책임이고, HttpRequest는 요청의 파싱 오류이기에 분리하는것이 바람직하지 않나 싶기도합니다.
확장성과 구조적인 측면, 그리고 책임을 기준으로 어떤 방식이 더 나을지, 아니면 이것보다 더 나은것은 없는지 확인해주세요.

선택한 이유:
현재 코드에 대한 설명, 나의 시도 및 고민, 판단 기준 등이 잘 드러나는 프롬프트라 생각한다.

### 7. 원본 프롬프트 구조 분석
- ✅문제 상황: ExceptionHandler(API 예외 처리)와 RequestHandler의 try-catch(파싱 예외 처리),
두 예외 처리 계층을 합칠지 분리 유지할지 고민.
- ✅원하는 결과를 어떻게 요구했는가: 구현이 아니라 "어느 구조가 나은지, 더 나은 대안은
없는지" 비교·판단을 요청함.
- 필요한 맥락이 포함되어 있는가: 위치와 역할은 설명했지만 코드는 안 보여줌. 실제론
바깥쪽 try가 Router.routing() 호출까지 감싸서 이미 경계가 겹친다는, 판단에 중요한
디테일이 빠짐.
- 판단 기준이나 제약 조건이 있는가: 확장성·구조·책임 세 기준은 있었지만 우선순위나
"기존 동작 유지" 같은 제약은 없음.
- 출력 방식이 명확한가: 불명확함. "확인해주세요"뿐, 표/결론/예시코드 등 원하는 형식을
지정 안 함.
- ✅나의 가설, 우려, 의도가 드러나는가: 가장 잘 드러남. 처음 생각(통합)과 재고(분리가
맞지 않나)가 다 담겨서, AI가 기각된 안을 다시 제안 안 하게 함.

### 8. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```
현재 예외 처리는 두 계층으로 나뉘어 있습니다. RequestHandler.run()의 try-catch는
HttpRequestParser.parse(in)뿐 아니라 Router.routing(request) 호출까지 같이 감쌉니다.
Router.routing() 내부의 ExceptionHandler는 API 실행 중 발생하는 예외를
상태 코드로 처리하고요. 그러니까 바깥쪽 try가 안쪽 호출까지 감싸고 있어서,
두 계층의 책임 경계가 코드상으로는 완전히 분리되어 있지 않습니다.

저는 처음엔 ExceptionHandler를 앞단으로 끌어올려 파싱 예외까지 한 번에 잡는 걸
생각했는데, 다시 보니 ExceptionHandler는 API 예외 처리, try-catch는 요청 자체의
파싱 검증이라는 서로 다른 책임이라 분리하는 게 맞지 않나 싶습니다.

책임 분리 > 구조적 명확성 > 확장성 순서로 기준을 두고, 지금처럼 분리 유지하는 안
(바깥쪽 try 범위를 파싱 호출로 좁히는 것 포함)과 제가 처음 생각한 통합안 중 뭐가
나은지, 더 나은 대안이 있다면 그것까지 포함해서 3~4줄로 짧게 비교하고 추천해주세요.
```

### 9. AI가 개선한 프롬프트 분석
- 예) 원본 프롬프트에 비해- 맥락/ 더 명확해진 목표/ 추가된 제약 조건/ 추가된 출력 형식/ 답변 품질이 좋아질 것으로 예상되는 이유 등

이전에 비해 응답으로 구체적인 줄수를 명시하고, 기준에 대해 순서를 지정해 주었다.
이렇게 한다면 확실히 더 우선순위가 높은 판단기준을 고려할 수 있을 것 같다.
그리고 구체적인 나의 안건 두개를 명시하여 AI가 더욱 확실한 선택을 할수 있도록 해주었다.

하지만 개인적인 생각으로는 코드에 대해는 AI가 자동으로 읽기에 자세히 설명하기 보단 문제에 대해 더 설명하는 방향으로 사용해보려한다.

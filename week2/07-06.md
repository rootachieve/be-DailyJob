### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - `Router`는 요청에 맞는 경로를 선택하는 책임만 지닌다.
  - `ExceptionHandler`는 `RequestHandler`에서 호출하지 않는다.
- 그렇게 판단한 이유:
  - 처음에는 AI 질문을 통해 직접 분리하였던 `ExceptionHandler`를 제거하고 예외 처리를 `Router` 내부에 두어 구조를 단순화하였다.
  - 하지만 이 경우 `Router`가 경로 선택뿐 아니라 예외의 종류를 판단하고 응답으로 변환하는 책임까지 가지게 되어 역할이 모호해진다 판단하였다.
  - 그렇기에 AI의 추천을 따르지 않고 분리 된 상태를 유지하도록 하였다.
  - 또한 `ExceptionHandler`를 `RequestHandler`에 두자는 제안을 예외가 뭉개진다 판단하여 추후 적용하기로 결정하였다.
- 고려했지만 선택하지 않은 대안:
  - `Router`에서 예외 처리를 하는 방법
  - 모든 예외를 `RequestHandler`에서 한 번에 처리하는 방법
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - `Router`는 라우팅 규칙에, `ExceptionHandler`는 예외와 HTTP 상태의 변환에 집중하게 되어 각 책임이 명확해졌다.
  - 새로운 `Route`나 예외 응답이 추가될 때 서로 다른 이유로 같은 클래스를 수정할 가능성을 줄였다.
  - 클래스와 호출 단계가 하나 늘어났기 때문에, 호출 흐름을 따라가는 것은 조금 더 복잡해졌지만, 반드시 필요한 분리 및 선택이라 생각한다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - `HttpRequest`에서 `POST` body를 읽는 `while`문에 들어가면 별다른 오류 메시지 없이 실행이 멈추었다.
- 처음 추측한 원인:
  - 개행의 문제라 생각하진 않았다. 이전 헤더를 읽는 과정에서 빈줄까지 확실히 읽어서 소비한다 생각하였기 때문이다.
  - body를 문자열로 변환하거나 스트림을 읽는 과정에서 예외가 발생해 요청 처리가 종료된다고 생각했다.
- 원인을 확인하기 위해 시도한 방법:
  - 요청 헤더와 body가 전달되는 흐름을 확인하고, `readLine()`의 종료 조건과 브라우저의 연결 유지 방식을 AI와 함께 점검했다.
  - 예외가 출력되지 않는 경로가 있는지도 확인했다.
- 실제 원인:
  - body의 끝에는 빈줄이 들어오지 않는다.
  - 그렇기에 `while`문이 탈출하지 못해 이전과 동일한 문제가 발생한 것이다.
- 해결 방법:
  - `Content-Length`를 기준으로 요청 body를 읽도록 변경하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 스트림을 읽을 때 프로토콜이 정의한 종료 조건이 무엇인지 먼저 확인한다.
  - 한 번의 `read()`로 필요한 데이터가 모두 들어온다고 가정하지 않고, 문자 수와 실제 바이트 수의 차이도 함께 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - AI가 제안한 구조를 그대로 확정하지 않고, `Router`의 책임이 모호해진다는 의문을 제기한 뒤 변경을 되돌리고 다시 책임을 나누었다.
  - 구현을 마친 뒤 `diff` 리뷰를 통해 구조, 동시성 등의 관점에서 놓친 문제를 추가로 확인했다.
- 그 접근이 도움이 된 이유:
  - AI의 답변에 의존하는대신 코드를 직접 읽고 구조를 이해하여 이후 작업시에도 잘못된 부분을 찾거나, 과한 요소를 제거할 수 있다.
  - `diff` 리뷰로 현재 코드의 퀄리티를 점검하고 부족한 부분을 수정하며 코드의 전반적인 퀄리티를 높일 수 있었다.
- 다음에도 반복하고 싶은 점:
  - AI의 제안도 하나의 대안으로 보고, 내가 생각한 책임과 변경 이유에 맞는지 다시 질문한다.
  - 주기적으로 `diff` 리뷰를 수행하여 코드의 퀄리티를 높여보려한다.

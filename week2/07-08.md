> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

  - 웹서버 6단계
    - HTML 요청도 API처럼 인가를 거치도록 서비스가 제공하는 화면을 모은 `PageApis` 생성
    - 로그인 상태에 따라 HTML 조각을 갈아끼우는 `DynamicHtml` 추가(마커 기반 split + 빌더/메서드 체이닝 구조)
    - `*`/`**` 와일드카드를 정규표현식으로 변환하는 방식으로 `Route`의 path 매칭 확장
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 학습
    - 바람직한 로깅 방법

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 화면들을 `PageApis`로 모아 기존 `Route`/세션 인가 구조를 그대로 재사용했다.
  - /index.html에 대한 처리를 와일드카드 path를 직접 구현해 해결하였다.
- 그렇게 판단한 이유:
  - 인가 구조를 새로 만들기보다 이미 검증된 `Route` 기반 인가를 페이지에도 그대로 적용하는 쪽이 책임을 덜 늘리는 선택이라 판단했다.
  - 다른 방식으로 html의 직접 접근을 막는것보다 와일드카드 path route로 범위를 통째로 막는게 효과적이라 판단했다.
- 고려했지만 선택하지 않은 대안:
  - `StaticFileHandler`에 인가 로직을 직접 추가하는 방법 -> 정적 파일 핸들러가 파일 응답과 인가 판단이라는 두 책임을 동시에 지게 되고, 향후 다른 리소스에 대한 접근 제어까지 전부 처리하긴 어렵다고 판단
  - `StaticFileHandler`에서 index.html을 필터링하는 방법 -> 과한 책임을 지니게 되고, 향후 놓치거나, 잊어버린 처리 공백이 생길수도 있다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 페이지 접근 제어가 기존 API 인가와 동일한 경로로 흐르게 되어, 새 페이지가 추가돼도 `PageApis`에 라우트만 등록하면 되는 구조가 되었다.
  - 와일드 카드 path로 다른 파일 추가시에도 등록이 간편해졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 서버는 로그인 상태를 알고 있지만 응답은 결국 순수 HTML이라, 같은 페이지에서 로그인/비로그인 상태에 따라 다른 내용을 어떻게 표현할지 막막했다.
- 처음 추측한 원인:
  - 순수 정적 HTML만으로 상태 분기를 표현할 방법 자체가 떠오르지 않는 상태였다.
- 원인을 확인하기 위해 시도한 방법:
  - 오래 고민하고, 다른 교육생에게 질문하여도 답이 보이지 않았다.
  - 결국 AI(클로드)에게 질문하였다.
- 실제 원인(막힌 지점):
  - "정적 HTML을 어떻게 동적으로 만들 것인가"라는 질문에 대한 구체적인 구현 아이디어가 없었던 것.
- 해결 방법:
  - HTML 안에 삽입 지점을 마커(`<!-- INSERT AUTH HEADER -->`)로 표시해두고, 서버가 로그인 상태에 맞는 HTML 조각으로 그 지점을 갈아끼우는 방식을 채택했다.
  - 이를 `DynamicHtml` 클래스로 구현하되, 향후 삽입 지점이 늘어날 것을 대비해 `setAuthHeader`처럼 메서드 체이닝으로 조립되는 구조로 만들었다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 클라이언트의 동적인 문제는 클라이언트에서 처리한다는 고정관념이 있었던 것 같다.
  - 다음에는 좀 더 다양한 시각에서 문제를 바라보고 해결해보려한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 어제, 그제 다짐한 대로 요구사항 분석과 고민의 흐름을 작업 전/중에 계속 메모하며 진행했다.
  - AI 제안을 받아들인 뒤에도 이에 대해 검증하 제안 이후에 생기는 후속 문제를 스스로 판단했다.
- 그 접근이 도움이 된 이유:
  - 메모 덕분에 오늘 이 회고와 PR 설명을 쓸 때 생각의 흐름을 다시 되짚지 않고도 바로 정리할 수 있었다.
  - AI 제안을 그 제안이 만들어내는 다음 문제까지 직접 추적함으로써, 예외 처리 위치를 구조에 맞게 스스로 결정할 수 있었다.
- 다음에도 반복하고 싶은 점:
  - 큰 구조 변경 전후로 생각을 메모하는 습관을 계속 유지한다.
  - AI 제안을 채택한 뒤에도 "이 제안이 만드는 효과는 무엇인가"를 한 단계 더 따라가 본다.

- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 오늘 채택한 아이디어 중 일부가(마커 기반 동적 HTML, 파일 접근 책임 분리)는 모두 AI 질문에 대한 답을 거의 그대로 받아들였다.
- 그렇게 된 원인:
  - "정적 HTML을 어떻게 동적으로 표현할까"라는 질문 자체가 이전에 명확한 후보가 없고, 생각하여도 떠오르지 않던 문제라 빠르게 채택했다.
- 다음에는 다르게 해볼 구체적인 행동:
  - AI의 답변이 바로 납득되더라도, 최소한 "이 방식 대신 템플릿 엔진 도입처럼 더 무거운 대안은 왜 아닌지"를 한 줄이라도 스스로 정리하고 넘어간다.
  - AI에게 답이 아닌 힌트를 달라고 하여 좀 더 생각해 볼 수 있는 기회를 가진다.

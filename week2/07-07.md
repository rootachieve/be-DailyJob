# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 웹서버 5단계
    - `HttpRequest`/`HttpResponse`에 쿠키 기능 추가
    - 인메모리 `SessionStorage` 추가 (UUID 세션 ID 생성, 만료시간 관리)
    - `Route`를 상속해 세션 인증을 확장한 `AuthRoute` 추가
    - `HttpResponse` 정적 팩토리 메서드에 헤더를 덧붙이는 `withHeader` 추가
    - 로그인 / 로그아웃 API 구현
  - 학습
    - 세션 저장 위치 비교(인메모리 vs DB)와 Redis가 세션 저장에 적합한 이유
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 학습
    - 스프링 시큐리티

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 인증이 필요한 `Route`만 세션 스토리지에서 `userId`를 조회해 넘겨받도록 `Route`를 상속하는 `AuthRoute`를 만들었다.
  - 이때 `User` 객체 전체가 아닌 `userId`(문자열)만 전달하도록 했다.
- 그렇게 판단한 이유:
  - `Router`가 쿠키를 확인해 `User`를 주입하는 방식은 요청마다 API를 다시 추가하는 구조가 되어버려 적절하지 않았다.
  - AI와 논의한 결과 `Router`는 API 라우터 탐색과 정적 파일 폴백(→404)에, `Route`는 조건과 처리에 집중해야 책임이 섞이지 않는다는 결론에 도달했다.
  - `User`를 그대로 넘기면 세션이 유지되는 동안 유저 정보가 바뀌는 상황에 대응할 수 없고, 비밀번호까지 함께 넘기게 되어 위험하다고 판단했다.
- 고려했지만 선택하지 않은 대안:
  - `ExceptionHandler` - 세션 조회 - `Router`를 메서드 체인처럼 연결하는 방식 -> 과하게 서로를 묶는 방식이라 판단
  - 세션에서 조회한 유저 정보를 전역 객체로 두고 참조하는 방식 -> 전역 객체에 접근하는 것이 구조상 좋지 않은 방식이라 생각함
  - `Router`가 쿠키를 직접 조회해 각 API에 `User`를 주입하는 방식 -> 라우터가 세션 스토리지를 알아야하고, 모든 API에 일관적으로 적용해야함
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 인증이 필요 없는 기존 `Route`는 전혀 영향을 받지 않고, 필요한 API만 `AuthRoute`로 확장해 사용할 수 있게 되었다.
  - `AuthRoute`를 세션 유무에 따라 독립적으로 테스트할 수 있게 되었다.
  - `Function` 대신 `BiFunction`을 사용하게 되어 시그니처는 조금 더 복잡해졌지만, 인증 정보가 전달되는 경로는 더 명확해졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 로그인과 회원가입의 바디 파싱을 정리하며, `BodyParser`를 제네릭으로 두고 `Map` 대신 바로 원하는 객체를 반환하도록 추상화하려 했다.
- 처음 추측한 원인:
  - 제네릭 타입 파라미터와 리플렉션을 함께 쓰면 파싱 결과를 곧바로 원하는 객체로 만들어낼 수 있으리라 생각했다.
- 원인을 확인하기 위해 시도한 방법:
  - 제네릭과 clsss 타입을 사용하여 리플렉션을 통해 객체를 생성해보려 시도하였다.
  - 인터페이스로 분리하고 UrlencodedBodyParser, JsonBodyParser와 같이 확장해보려 하였다.
- 실제 원인:
  - 하지만 실제 코드를 작성하자 리플렉션은 내가 생각했던 것보다 복잡하고, 처리하기 어려운 방식이였다.
  - 또한 Json같은 경우는 이후 역직렬화 라이브러리를 사용하면 간단하게 할 수 있다.
- 해결 방법:
  - 제네릭 + 리플렉션 접근을 포기하였다.
  - 인터페이스와 구현체로 분리하는 방안도 검토했지만 지금 시점엔 그마저 과하다 판단해, `Map`을 반환하는 단순한 `UrlencodedBodyParser` 클래스로 정리했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 작업 수행 이전 미래에 생각하는 흐름을 ai에게 설명하여 이 방향이 괜찮은지 한번더 점검 받아, 불필요한 구현을 하지 않으려한다.
  - 간단하게 할 수 있는 방법을 먼저 생각하고, 적용해보려한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 어제 회고에서 다짐한 대로, 요구사항을 분석하고 구조를 잡아나가는 과정의 생각 흐름을 계속 메모하며 진행했다.
  - AI가 제안한 구조(체인 연결, `User` 직접 전달 등)를 바로 받아들이지 않고 "책임이 섞이는 게 아닌가"를 되물으며 대안을 함께 검토했다.
- 그 접근이 도움이 된 이유:
  - 메모 덕분에 왜 `AuthRoute`와 `userId` 전달 방식을 선택했는지 근거가 남았고, 이후 로그인과 로그아웃 구현 단계에서 헤매지 않고 이어갈 수 있었다.
  - AI 제안을 검증하는 과정에서 `Router`와 `Route`의 책임 경계를 스스로 더 명확하게 정리할 수 있었다.
- 다음에도 반복하고 싶은 점:
  - 기능 구현 전 구조를 고민하는 단계에서 생각을 메모로 남기는 것을 계속 습관화한다.
  - AI 제안은 채택 전에 반드시 내 판단 기준(책임 분리, 보안 등)으로 한 번 더 검증한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - `BodyParser`를 제네릭과 리플렉션으로 일반화하려다, 실현 가능성을 미리 확인하지 않고 바로 구현에 들어가 결국 쓰지 못하게 되는 코드에 시간을 많이 썼다.
- 그렇게 된 원인:
  - 현재 서비스에 당장 필요한 수준(단순 `Map` 파싱)보다 앞서, "이왕이면 확장 가능하게" 만들고 싶어 필요 이상의 추상화를 먼저 시도했다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 이전에도 Accept 헤더 처리에서 구현 비용을 미리 따지지 못해 비슷한 시행착오를 겪었는데, 오늘도 비슷한 상황인 듯 하다.
  - 새로운 추상화를 시도하기 전에 난이도와 그 구현으로 얻는 이득을 검증하고, 지금 필요한 최소 기능부터 구현한 뒤 필요해지면 확장하는 순서를 지키려 한다.
